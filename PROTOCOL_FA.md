# پروتکل رمزنگاری Lapka SMS

نسخه: **3** (`psms-lib`)

این سند پروتکل رمزنگاری مورد استفاده در Lapka SMS را برای محافظت از محتوای پیام‌های SMS توصیف می‌کند. این پروتکل رمزنگاری احراز هویت‌شده با محافظت از بازپخش و کدگذاری پنهان‌نگاری را ارائه می‌دهد.

## نمای کلی

```
متن ساده
   │
   ▼
┌─────────────────────┐
│  Plain Data Encoder  │  فشرده‌سازی متن به بایت (بهترین کدگذار خودکار انتخاب می‌شود)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│       Pack          │  اضافه کردن channel ID (اختیاری) + بایت MetaInfo
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│    AES-256-GCM      │  رمزنگاری با کلید مشتق‌شده؛ nonce = timestamp || random
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Encrypted Data      │  کدگذاری متن رمز به Base64 / Base64 سیریلیک /
│ Encoder             │  کلمات روسی
└──────────┬──────────┘
           ▼
      متن SMS
```

## استخراج کلید

یک **کلید اصلی** واحد (۱۲۸، ۱۹۲ یا ۲۵۶ بیت) بین طرفین به اشتراک گذاشته می‌شود. یک زیرکلید رمزنگاری با استفاده از HKDF (RFC 5869) با HMAC-SHA256 مشتق می‌شود:

```
HKDF-Extract:
  salt = "k-sms-hkdf-v2" (ثابت، عمومی)
  IKM  = master_key
  PRK  = HMAC-SHA256(salt, IKM)

HKDF-Expand (کلید رمزنگاری):
  info   = "k-sms-v2-enc"
  length = 32 بایت
  enc_key = HKDF-Expand(PRK, info, 32)
```

رشته‌های salt و info ثابت‌های پروتکل هستند، نه اسرار. HKDF جداسازی دامنه را فراهم می‌کند و توزیع یکنواخت کلید رمزنگاری را صرف‌نظر از کیفیت کلید اصلی تضمین می‌کند.

## کدگذاری داده‌های ساده

قبل از رمزنگاری، متن پیام با استفاده از فشرده‌ترین کدگذار موجود به آرایه بایت فشرده می‌شود. کدگذار به طور خودکار بر اساس محتوای پیام انتخاب می‌شود:

| حالت | شناسه | توصیف |
|---|---|---|
| SHORT_CP1251_PREFER_CYRILLIC | 0 | کدگذاری فشرده CP1251، بهینه‌شده برای سیریلیک |
| SHORT_CP1251_PREFER_LATIN | 1 | کدگذاری فشرده CP1251، بهینه‌شده برای لاتین |
| CP1251 | 2 | کدگذاری استاندارد CP1251 |
| UTF_8 | 3 | UTF-8 استاندارد (پشتیبان برای هر متنی) |
| ASCII | 4 | ASCII ۷ بیتی |
| HUFFMAN_CYRILLIC | 5 | کد هافمن بهینه‌شده برای سیریلیک |
| HUFFMAN_LATIN | 6 | کد هافمن بهینه‌شده برای لاتین |

شناسه کدگذار در بایت MetaInfo ذخیره می‌شود تا گیرنده بداند چگونه رمزگشایی کند.

## بسته‌بندی پیام

بایت‌های کدگذاری‌شده متن با حداقل فراداده در payload بسته‌بندی می‌شوند:

```
┌──────────────┬─────────────────┬──────────┐
│ encoded_text │ channel_id (4B) │ meta (1B)│
│ (متغیر)     │ (اختیاری)       │          │
└──────────────┴─────────────────┴──────────┘
```

### فیلدها

**encoded_text** (طول متغیر): متن پیام فشرده‌شده توسط کدگذار.

**channel_id** (۴ بایت، اختیاری): شناسه کانال مکالمه. فقط در صورت تنظیم پرچم `isChannel` در MetaInfo موجود است. uint32 به صورت Little-endian.

**MetaInfo** (۱ بایت): بایت فراداده بیت‌فشرده:

```
چیدمان بیت‌ها: [C][VVV][MMMM]

بیت‌های 0-3 (MMMM): حالت کدگذار متن ساده (0-6)
بیت‌های 4-6 (VVV):   نسخه پروتکل (فعلاً 3)
بیت 7 (C):           پرچم وجود channel ID (0 = خیر، 1 = بله)
```

بدون padding، بدون HMAC، بدون timestamp در payload. تایم‌استمپ در GCM nonce حمل می‌شود. احراز هویت توسط تگ GCM فراهم می‌شود.

## رمزنگاری

payload بسته‌بندی‌شده با **AES-256-GCM** و **تگ احراز هویت ۱۲۸ بیتی** رمزنگاری می‌شود:

```
timestamp = زمان فعلی یونیکس (4 بایت، little-endian)
random    = SecureRandom(8 بایت)
nonce     = timestamp || random     (مجموعاً 12 بایت)

ciphertext || tag = AES-256-GCM(enc_key, nonce, payload)
                    تگ = 128 بیت (16 بایت)
```

فرمت انتقال:

```
┌────────────────────────────┬────────────────────────────────┐
│        nonce (12B)         │  ciphertext + تگ GCM (16B)     │
│ [timestamp 4B][random 8B] │                                │
└────────────────────────────┴────────────────────────────────┘
```

### ویژگی‌ها

- **ساختار nonce**: مجموعاً ۱۲ بایت — ۴ بایت اول تایم‌استمپ یونیکس (little-endian)، ۸ بایت آخر تصادفی از `SecureRandom`. این ۲⁶⁴ فضای تصادفی در ثانیه فراهم می‌کند و برخورد nonce را بسیار بعید می‌سازد (~۵۸ میلیون سال با ۱۰۰۰ پیام/روز).
- **تگ GCM**: ۱۲۸ بیت (۱۶ بایت) — حداکثر مجاز توسط NIST SP 800-38D. احتمال جعل: ۲⁻¹²⁸ در هر تلاش.
- **رمز**: `AES/GCM/NoPadding` (javax.crypto، شتاب سخت‌افزاری در اندروید). GCM داخلاً در حالت CTR کار می‌کند، بنابراین padding بلوک لازم نیست.
- **تایم‌استمپ در nonce**: nonce به صورت متن ساده (قبل از متن رمز) منتقل می‌شود. این به گیرنده اجازه می‌دهد تازگی پیام را *قبل* از انجام رمزگشایی GCM پرهزینه بررسی کند. **مصالحه**: از آنجا که اعتبارسنجی تایم‌استمپ قبل از احراز هویت GCM رخ می‌دهد، مهاجم می‌تواند بفهمد آیا پیام جعلی از بررسی تایم‌استمپ عبور کرده است. این مصالحه قابل قبولی برای SMS است — فقط فاش می‌کند که تایم‌استمپ جعلی در پنجره ۷ روزه قرار دارد. تگ GCM همچنان کاملاً از جعل محافظت می‌کند.
- **بدون HMAC جداگانه**: GCM یک رمز AEAD است — تگ GCM از قبل هم یکپارچگی و هم اصالت کل payload را فراهم می‌کند.
- **بدون padding PKCS#7**: GCM/CTR یک حالت رمز جریانی است و طول‌های دلخواه متن ساده را مدیریت می‌کند.

### سربار

سربار ثابت هر پیام: **۲۹ بایت** (۱۲ بایت nonce + ۱۶ بایت تگ GCM + ۱ بایت MetaInfo).

## کدگذاری داده‌های رمزنگاری‌شده (پنهان‌نگاری)

آرایه بایت رمزنگاری‌شده برای انتقال از طریق SMS به رشته متنی کدگذاری می‌شود. سه طرح موجود است:

### Base64 (طرح 0)

کدگذاری Base64 استاندارد. فشرده اما از نظر بصری به عنوان داده کدگذاری‌شده آشکار است.

```
ورودی:  [0xDE, 0xAD, ...]
خروجی: "3q0t..."
```

### Base64 سیریلیک (طرح 1)

Base64 با جایگزینی الفبای لاتین با کاراکترهای سیریلیک:

```
لاتین:    ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/=
سیریلیک: АБВГДЕЁЖЗИЙКЛМНОПРСТУФХЦЧШЩЫЬЭЮЯабвгдеёжзийклмнопрстуфхцчшщъыьэюя
```

خروجی برای ناظر عادی مانند متن سیریلیک به نظر می‌رسد.

### کلمات روسی / متن (طرح 2)

بایت‌های رمزنگاری‌شده را به عنوان متن روسی طبیعی با استفاده از سیستمی از زیرکدگذارها کدگذاری می‌کند:

- **زیرکدگذارهای لغت‌نامه**: مقادیر بایت را به کلمات روسی از لغت‌نامه‌ها نگاشت می‌کنند
- **زیرکدگذار تاریخ/زمان**: مقادیر را به رشته‌های تاریخ/زمان کدگذاری می‌کند
- **زیرکدگذار نقطه‌گذاری**: مقادیر را به علائم نگارشی کدگذاری می‌کند

خروجی مانند دنباله‌ای از کلمات روسی با نقطه‌گذاری و فاصله‌گذاری طبیعی به نظر می‌رسد.

## محافظت از بازپخش

محافظت از بازپخش از دو مکانیزم مستقل استفاده می‌کند:

### ۱. اعتبارسنجی تایم‌استمپ

هر پیام یک تایم‌استمپ یونیکس ۴ بایتی حمل می‌کند که در ۴ بایت اول GCM nonce تعبیه شده است. هنگام رمزگشایی، تایم‌استمپ استخراج و **قبل** از رمزگشایی GCM اعتبارسنجی می‌شود:

- پیام‌های قدیمی‌تر از **۷ روز** رد می‌شوند
- پیام‌هایی که بیش از **۱۰ دقیقه** در آینده هستند رد می‌شوند

این امکان رد سریع پیام‌های قدیمی بدون هیچ سربار رمزنگاری را فراهم می‌کند.

### ۲. حافظه نهان nonce

پس از رمزگشایی موفق GCM (اثبات اصالت پیام)، nonce ۱۲ بایتی با حافظه نهان درون‌حافظه‌ای nonce‌های اخیراً دیده‌شده مقایسه می‌شود. اگر nonce قبلاً دیده شده باشد، پیام به عنوان بازپخش رد می‌شود.

ویژگی‌های حافظه نهان:
- **اندازه**: تا ۱۰۰۰ ورودی (~۱۲ کیلوبایت حافظه)
- **TTL**: ۷ روز — ورودی‌های قدیمی‌تر از ۷ روز به طور خودکار حذف می‌شوند
- **حذف**: بر اساس سن (TTL) و FIFO هنگام تجاوز از ظرفیت
- **پایداری**: فقط درون‌حافظه‌ای (با راه‌اندازی مجدد برنامه پاک می‌شود)
- **امنیت نخ**: دسترسی همگام‌سازی‌شده

## خط لوله رمزگشایی

```
متن SMS
   │
   ▼
┌─────────────────────┐
│ Encrypted Data      │  رمزگشایی از Base64/سیریلیک/کلمات روسی
│ Decoder             │  (حذف padding جلو در صورت نیاز)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ استخراج nonce       │  خواندن ۱۲ بایت اول؛ استخراج تایم‌استمپ از بایت‌های 0-3
│ اعتبارسنجی timestamp│  رد اگر خارج از پنجره ۷ روزه (قبل از رمزگشایی GCM)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ AES-256-GCM decrypt │  تأیید تگ ۱۲۸ بیتی + رمزگشایی payload
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ بررسی حافظه nonce   │  رد اگر nonce قبلاً دیده شده (ضد بازپخش)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ باز کردن بسته       │  تجزیه MetaInfo، استخراج channel ID، بایت‌های متن
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Plain Data Decoder  │  رفع فشرده‌سازی بایت‌های متن بر اساس حالت MetaInfo
└──────────┬──────────┘
           ▼
      متن ساده
```

## محافظت smsForReset

ویژگی «SMS برای بازنشانی» اجازه می‌دهد یک عبارت ماشه خاص از راه دور همه کلیدهای رمزنگاری را پاک کند. این با مقایسه مبتنی بر هش محافظت می‌شود:

۱. هنگامی که کاربر عبارت ماشه را تنظیم می‌کند، هم عبارت (در EncryptedSharedPreferences) و هم **هش SHA-256** آن ذخیره می‌شوند
۲. هنگام رمزگشایی پیام ورودی، متن آن با SHA-256 هش می‌شود
۳. هش با هش ذخیره‌شده با استفاده از **مقایسه زمان‌ثابت** (`MessageDigest.isEqual()`) مقایسه می‌شود

## ذخیره‌سازی کلید (اندروید)

### کلیدهای رمزنگاری پیام

کلیدهای اصلی برای رمزنگاری پیام با استفاده از **EncryptedSharedPreferences** (AndroidX Security) ذخیره می‌شوند:

- کلید اصلی: AES-256-GCM (تولیدشده از طریق Android Keystore)
- رمزنگاری کلید: AES-256-SIV
- رمزنگاری مقدار: AES-256-GCM

کلیدها می‌توانند به صورت سراسری (برای همه مکالمات) یا برای هر مکالمه (ذخیره‌شده در پایگاه داده رمزنگاری‌شده Realm) تنظیم شوند.

### کلید پایگاه داده Realm

پایگاه داده Realm با کلید ۵۱۲ بیتی رمزنگاری شده است:

۱. یک کلید تصادفی ۶۴ بایتی از طریق `SecureRandom` تولید می‌شود
۲. یک کلید AES-256-GCM در **Android Keystore** (TEE سخت‌افزاری) ایجاد می‌شود
۳. کلید Realm با کلید Keystore رمزنگاری می‌شود
۴. کلید رمزنگاری‌شده Realm و IV در SharedPreferences ذخیره می‌شوند
۵. در هر راه‌اندازی برنامه، کلید Realm با کلید Keystore رمزگشایی می‌شود

```
┌────────────┐     AES-256-GCM      ┌─────────────────────┐
│  Android   │ ───────────────────►  │  SharedPreferences   │
│  Keystore  │   encrypt(realm_key)  │  (کلید رمز + IV)     │
│  (TEE)     │ ◄─────────────────── │                      │
│            │   decrypt(realm_key)  │                      │
└────────────┘                       └─────────────────────┘
```

## تبادل کلید

کلیدها **خارج از باند** بین کاربران تبادل می‌شوند:

۱. **QR کد**: یک کاربر QR کد حاوی کلید Base64-کدگذاری‌شده تولید می‌کند؛ دیگری آن را اسکن می‌کند
۲. **ورود دستی**: کاربران کلید Base64-کدگذاری‌شده را کپی/پیست می‌کنند

اصالت کلید با مقایسه **اثر انگشت SHA-256** (۱۶ بایت اول، نمایش‌داده‌شده به صورت ایموجی) قابل تأیید است:

```
بایت‌های کلید → SHA-256 → ۱۶ بایت اول → هر بایت به یکی از ۲۵۶ ایموجی منحصربه‌فرد نگاشت می‌شود
در ۴ ردیف ۴ ایموجی برای مقایسه بصری آسان نمایش داده می‌شود
```

هر دو طرف باید اثر انگشت را از طریق کانال امن جداگانه تأیید کنند.

## ویژگی‌های امنیتی

| ویژگی | مکانیزم |
|---|---|
| محرمانگی | AES-256-GCM |
| یکپارچگی و اصالت | تگ احراز هویت GCM (۱۲۸ بیت) |
| محافظت از بازپخش | تایم‌استمپ در nonce (پنجره ۷ روزه) + حافظه نهان nonce (۱۰۰۰ ورودی، TTL ۷ روز) |
| جداسازی کلید | HKDF با رشته info خاص دامنه |
| ذخیره‌سازی کلید | Android Keystore (TEE) + EncryptedSharedPreferences |
| پنهان‌نگاری | کدگذاری Base64 / Base64 سیریلیک / کلمات روسی |
| مقاومت در برابر حمله زمان‌بندی | مقایسه هش زمان‌ثابت برای smsForReset |
| مقاومت در برابر تکرار nonce | timestamp(4B) + random(8B) = ۲⁶⁴ فضای تصادفی در ثانیه |

## خلاصه فرمت انتقال

```
روی سیم (بعد از کدگذاری پنهان‌نگاری):

┌────────────────────────────┬────────────────────────────────┐
│        nonce (12B)         │  ciphertext + تگ GCM            │
│ [timestamp 4B][random 8B] │  (تگ = 16B)                     │
└────────────────────────────┴────────────────────────────────┘

داخل ciphertext (بعد از رمزگشایی GCM):

┌──────────────┬─────────────────┬──────────┐
│ encoded_text │ channel_id (4B) │ meta (1B)│
│ (متغیر)     │ (اختیاری)       │          │
└──────────────┴─────────────────┴──────────┘

سربار ثابت: ۲۹ بایت (12B nonce + 16B تگ GCM + 1B MetaInfo)
```

## محدودیت‌ها

- **بدون forward secrecy**: به خطر افتادن کلید اصلی همه پیام‌های گذشته و آینده را رمزگشایی می‌کند
- **بدون key ratcheting**: همان کلید اصلی برای همه پیام‌ها در یک مکالمه استفاده می‌شود
- **دقت تایم‌استمپ**: دقت ۱ ثانیه، تایم‌استمپ یونیکس ۳۲ بیتی (سرریز در سال ۲۱۰۶)
- **محدودیت‌های اندازه SMS**: کدگذاری پنهان‌نگاری اندازه پیام را افزایش می‌دهد؛ پیام‌های طولانی ممکن است توسط اپراتور به چند بخش SMS تقسیم شوند
- **بدون قابلیت انکار**: هر دو طرف کلید متقارن یکسانی دارند و می‌توانند ثابت کنند طرف دیگر پیام را ارسال کرده
- **حافظه نهان nonce درون‌حافظه‌ای**: تشخیص بازپخش با راه‌اندازی مجدد برنامه از بین می‌رود؛ در محدوده پنجره ۷ روزه تایم‌استمپ، راه‌اندازی مجدد اجازه بازپخش پیام‌های قبلاً دیده‌شده را می‌دهد
