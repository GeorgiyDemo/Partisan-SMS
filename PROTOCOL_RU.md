# Протокол шифрования Lapka SMS

Версия: **3** (`psms-lib`)

Этот документ описывает протокол шифрования, используемый в Lapka SMS для защиты содержимого SMS-сообщений. Протокол обеспечивает аутентифицированное шифрование с защитой от replay-атак и стеганографическим кодированием.

## Общая схема

```
Открытый текст
   │
   ▼
┌─────────────────────┐
│  Plain Data Encoder  │  Сжатие текста в байты (лучший кодировщик выбирается автоматически)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│       Pack          │  Добавление channel ID (опционально) + MetaInfo
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│    AES-256-GCM      │  Шифрование производным ключом; nonce = timestamp || random
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Encrypted Data      │  Кодирование шифротекста в Base64 / Кириллический Base64 /
│ Encoder             │  Русские слова
└──────────┬──────────┘
           ▼
      Текст SMS
```

## Вывод ключей

Единый **мастер-ключ** (128, 192 или 256 бит) разделяется между сторонами. Подключ шифрования выводится через HKDF (RFC 5869) с HMAC-SHA256:

```
HKDF-Extract:
  salt = "k-sms-hkdf-v2" (фиксированная, публичная)
  IKM  = master_key
  PRK  = HMAC-SHA256(salt, IKM)

HKDF-Expand (ключ шифрования):
  info   = "k-sms-v2-enc"
  length = 32 байта
  enc_key = HKDF-Expand(PRK, info, 32)
```

Salt и info — фиксированные константы протокола, не секреты. HKDF обеспечивает доменное разделение и гарантирует равномерное распределение ключа шифрования вне зависимости от качества мастер-ключа.

## Кодирование открытого текста

Перед шифрованием текст сообщения сжимается в массив байтов с помощью наиболее компактного доступного кодировщика. Кодировщик выбирается автоматически на основе содержимого:

| Режим | ID | Описание |
|---|---|---|
| SHORT_CP1251_PREFER_CYRILLIC | 0 | Компактная CP1251, оптимизирована для кириллицы |
| SHORT_CP1251_PREFER_LATIN | 1 | Компактная CP1251, оптимизирована для латиницы |
| CP1251 | 2 | Стандартная CP1251 |
| UTF_8 | 3 | Стандартный UTF-8 (запасной для любого текста) |
| ASCII | 4 | 7-битный ASCII |
| HUFFMAN_CYRILLIC | 5 | Код Хаффмана, оптимизирован для кириллицы |
| HUFFMAN_LATIN | 6 | Код Хаффмана, оптимизирован для латиницы |

ID кодировщика сохраняется в байте MetaInfo (см. ниже), чтобы получатель знал, как декодировать.

## Упаковка сообщения

Закодированные байты текста упаковываются в payload с минимальными метаданными:

```
┌──────────────┬─────────────────┬──────────┐
│ encoded_text │ channel_id (4Б) │ meta (1Б)│
│ (переменная) │ (опционально)   │          │
└──────────────┴─────────────────┴──────────┘
```

### Поля

**encoded_text** (переменная длина): Текст сообщения, сжатый кодировщиком.

**channel_id** (4 байта, опционально): Идентификатор канала диалога. Присутствует только если установлен флаг `isChannel` в MetaInfo. Little-endian uint32.

**MetaInfo** (1 байт): Битово-упакованный байт метаданных:

```
Раскладка битов: [C][VVV][MMMM]

Биты 0-3 (MMMM): Режим кодировщика текста (0-6)
Биты 4-6 (VVV):   Версия протокола (сейчас 3)
Бит 7 (C):        Флаг наличия channel ID (0 = нет, 1 = да)
```

Без паддинга, без HMAC, без timestamp в payload. Timestamp несётся в GCM nonce (см. ниже). Аутентификация обеспечивается GCM-тегом.

## Шифрование

Упакованный payload шифруется **AES-256-GCM** с **128-битным тегом аутентификации**:

```
timestamp = текущее Unix-время (4 байта, little-endian)
random    = SecureRandom(8 байт)
nonce     = timestamp || random     (12 байт всего)

ciphertext || tag = AES-256-GCM(enc_key, nonce, payload)
                    тег = 128 бит (16 байт)
```

Формат на проводе:

```
┌────────────────────────────┬────────────────────────────────┐
│        nonce (12Б)         │  ciphertext + GCM тег (16Б)    │
│ [timestamp 4Б][random 8Б] │                                │
└────────────────────────────┴────────────────────────────────┘
```

### Свойства

- **Структура nonce**: 12 байт — первые 4 байта это Unix timestamp (little-endian), последние 8 байт случайны через `SecureRandom`. Это обеспечивает 2⁶⁴ случайного пространства в секунду, делая коллизию nonce астрономически маловероятной (~58 млн лет при 1000 сообщений/день).
- **GCM-тег**: 128 бит (16 байт) — максимальный размер по NIST SP 800-38D. Вероятность подделки: 2⁻¹²⁸ на попытку.
- **Шифр**: `AES/GCM/NoPadding` (javax.crypto, аппаратно ускоренный на Android). GCM работает внутри в режиме CTR, поэтому блочный паддинг не нужен — произвольная длина plaintext обрабатывается нативно.
- **Timestamp в nonce**: Nonce передаётся открытым текстом (перед шифротекстом). Это позволяет получателю проверить свежесть сообщения *до* выполнения дорогой GCM-расшифровки, обеспечивая раннее отклонение устаревших/повторных сообщений при почти нулевых CPU-затратах. **Компромисс**: Поскольку проверка timestamp происходит до GCM-аутентификации, атакующий может узнать, прошло ли поддельное сообщение проверку timestamp (через разницу во времени обработки). Это приемлемый компромисс для SMS — он лишь раскрывает, что поддельный timestamp попадает в 7-дневное окно, что и так тривиально угадать. GCM-тег по-прежнему полностью защищает от подделки.
- **Без отдельного HMAC**: GCM — это AEAD-шифр (Authenticated Encryption with Associated Data). GCM-тег уже обеспечивает и целостность, и аутентичность всего payload. Отдельный HMAC был бы избыточен.
- **Без PKCS#7-паддинга**: GCM/CTR — потоковый режим шифрования, обрабатывающий произвольную длину plaintext. Блочный паддинг не нужен и тратил бы байты впустую.

### Overhead

Фиксированный overhead на сообщение: **29 байт** (12Б nonce + 16Б GCM-тег + 1Б MetaInfo).

## Кодирование шифротекста (стеганография)

Зашифрованный массив байтов кодируется в текстовую строку для передачи через SMS. Доступны три схемы:

### Base64 (Схема 0)

Стандартное Base64-кодирование. Компактно, но визуально очевидно как закодированные данные.

```
Вход:  [0xDE, 0xAD, ...]
Выход: "3q0t..."
```

### Кириллический Base64 (Схема 1)

Base64, где латинский алфавит заменён кириллическими символами:

```
Латиница:  ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/=
Кириллица: АБВГДЕЁЖЗИЙКЛМНОПРСТУФХЦЧШЩЫЬЭЮЯабвгдеёжзийклмнопрстуфхцчшщъыьэюя
```

Вывод выглядит как кириллический текст для случайного наблюдателя.

### Русские слова / Текст (Схема 2)

Кодирует зашифрованные байты как естественно выглядящий русский текст с помощью системы суб-кодировщиков:

- **Словарные суб-кодировщики**: Отображают значения байтов в русские слова из словарей
- **DateTime суб-кодировщик**: Кодирует значения как строки даты/времени
- **Суб-кодировщик пунктуации**: Кодирует значения как знаки препинания

Кодирование трактует зашифрованные данные как большое целое число и выполняет разложение в смешанной системе счисления по доступным суб-кодировщикам. Случайный байт переднего паддинга добавляется при необходимости, чтобы число корректно отображалось в доступное пространство слов.

Вывод выглядит как последовательность русских слов с естественной пунктуацией и пробелами. При декодировании удаляется до 256 байтов переднего паддинга для нахождения правильного выравнивания.

## Защита от replay-атак

Используются два независимых механизма:

### 1. Валидация timestamp

Каждое сообщение несёт 4-байтовый Unix timestamp, встроенный в первые 4 байта GCM nonce. При расшифровке timestamp извлекается и проверяется **до** GCM-расшифровки:

- Сообщения старше **7 дней** отклоняются
- Сообщения из будущего более чем на **10 минут** отклоняются

Это обеспечивает быстрое отклонение устаревших сообщений без каких-либо криптографических затрат.

### 2. Кеш nonce

После успешной GCM-расшифровки (подтверждающей подлинность сообщения), 12-байтный nonce проверяется по in-memory кешу недавно виденных nonce. Если nonce уже был виден, сообщение отклоняется как replay.

Кеш проверяется *после* GCM-аутентификации для предотвращения отравления кеша — атакующий не может засорить кеш фейковыми nonce, так как поддельные сообщения провалят GCM-аутентификацию до проверки кеша.

Свойства кеша:
- **Размер**: До 1000 записей (~12 КБ памяти)
- **TTL**: 7 дней — записи старше 7 дней удаляются автоматически
- **Вытеснение**: По возрасту (TTL) и FIFO при превышении ёмкости
- **Персистентность**: Только в памяти (очищается при перезапуске приложения)
- **Потокобезопасность**: Синхронизированный доступ

## Конвейер расшифровки

```
Текст SMS
   │
   ▼
┌─────────────────────┐
│ Encrypted Data      │  Декодирование из Base64/Кириллицы/Русских слов
│ Decoder             │  (удаление переднего паддинга при необходимости)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Извлечение nonce    │  Чтение первых 12 байт; извлечение timestamp из байт 0-3
│ Валидация timestamp │  Отклонение если вне 24-часового окна (до GCM-расшифровки)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ AES-256-GCM decrypt │  Проверка 128-битного тега + расшифровка payload
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Проверка кеша nonce │  Отклонение если nonce уже был виден (anti-replay)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Распаковка          │  Разбор MetaInfo, извлечение channel ID, байтов текста
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Plain Data Decoder  │  Декомпрессия байтов текста на основе режима из MetaInfo
└──────────┬──────────┘
           ▼
      Открытый текст
```

## Защита smsForReset

Функция «SMS для сброса» позволяет специальной триггер-фразе удалённо стереть все ключи шифрования. Это защищено хеш-сравнением:

1. Когда пользователь устанавливает триггер-фразу, сохраняются и фраза (в EncryptedSharedPreferences), и её **SHA-256 хеш**
2. При расшифровке входящего сообщения его текст хешируется SHA-256
3. Хеш сравнивается с сохранённым хешем через **constant-time сравнение** (`MessageDigest.isEqual()`)

Это обеспечивает defense-in-depth: даже если EncryptedSharedPreferences каким-то образом будут скомпрометированы, операция сравнения не утекает информацию о времени (timing).

## Хранение ключей (Android)

### Ключи шифрования сообщений

Мастер-ключи для шифрования сообщений хранятся через **EncryptedSharedPreferences** (AndroidX Security):

- Мастер-ключ: AES-256-GCM (создаётся через Android Keystore)
- Шифрование ключей: AES-256-SIV
- Шифрование значений: AES-256-GCM

Ключи могут быть установлены глобально (для всех диалогов) или для каждого диалога отдельно (хранятся в зашифрованной базе Realm).

### Ключ базы данных Realm

База данных Realm зашифрована 512-битным ключом:

1. Генерируется случайный 64-байтовый ключ через `SecureRandom`
2. В **Android Keystore** (аппаратный TEE) создаётся ключ AES-256-GCM
3. Ключ Realm шифруется ключом из Keystore
4. Зашифрованный ключ Realm и IV сохраняются в SharedPreferences
5. При каждом запуске приложения ключ Realm расшифровывается ключом из Keystore

```
┌────────────┐     AES-256-GCM      ┌─────────────────────┐
│  Android   │ ───────────────────►  │  SharedPreferences   │
│  Keystore  │   encrypt(realm_key)  │  (зашифр. ключ + IV) │
│  (TEE)     │ ◄─────────────────── │                      │
│            │   decrypt(realm_key)  │                      │
└────────────┘                       └─────────────────────┘
```

## Обмен ключами

Ключи обмениваются **вне основного канала** между пользователями:

1. **QR-код**: Один пользователь генерирует QR-код с Base64-кодированным ключом; второй сканирует его
2. **Ручной ввод**: Пользователи копируют/вставляют Base64-кодированный ключ

Подлинность ключа можно проверить сравнением **SHA-256 отпечатков** (первые 16 байт, отображаемые как эмодзи):

```
Байты ключа → SHA-256 → первые 16 байт → каждый байт отображается в один из 256 уникальных эмодзи
Отображается в виде 4 строк по 4 эмодзи для удобного визуального сравнения
```

Обе стороны должны сверить отпечаток через отдельный защищённый канал.

## Свойства безопасности

| Свойство | Механизм |
|---|---|
| Конфиденциальность | AES-256-GCM |
| Целостность и аутентичность | GCM-тег (128 бит) |
| Защита от replay | Timestamp в nonce (7-дневное окно) + кеш nonce (1000 записей, TTL 7 дней) |
| Разделение ключей | HKDF с доменной info-строкой |
| Хранение ключей | Android Keystore (TEE) + EncryptedSharedPreferences |
| Стеганография | Base64 / Кириллический Base64 / Русские слова |
| Защита от timing-атак | Constant-time сравнение хеша для smsForReset |
| Защита от повтора nonce | timestamp(4Б) + random(8Б) = 2⁶⁴ случайного пространства в секунду |

## Формат на проводе

```

│ [timestamp 4Б][random 8Б] │  (тег = 16Б)                   │

Внутри ciphertext (после GCM-расшифровки):

┌──────────────┬─────────────────┬──────────┐

│ (переменная) │ (опционально)   │          │

Фиксированный overhead: 29 байт (12Б nonce + 16Б GCM-тег + 1Б MetaInfo)
```

## Ограничения

- **Нет forward secrecy**: Компрометация мастер-ключа раскрывает все прошлые и будущие сообщения
- **Нет ротации ключей (ratcheting)**: Один мастер-ключ используется для всех сообщений в диалоге
- **Точность timestamp**: Разрешение 1 секунда, 32-битный Unix timestamp (переполнение в 2106 году)
- **Ограничения размера SMS**: Стеганографическое кодирование увеличивает размер сообщения; длинные сообщения могут разбиваться на несколько SMS-сегментов оператором
- **Нет отрицаемости**: Обе стороны используют один симметричный ключ и могут доказать, что другая сторона отправила сообщение
- **Кеш nonce в памяти**: Защита от replay теряется при перезапуске приложения; в пределах 7-дневного окна timestamp перезапуск позволяет повтор ранее виденных сообщений
