# Протокол шифрования Lapka SMS

Версия: **3** (`psms-lib`)

Этот документ описывает протокол шифрования, используемый в Lapka SMS для защиты содержимого SMS-сообщений. Протокол обеспечивает аутентифицированное шифрование с защитой от replay-атак и стеганографическим кодированием.

## Общая схема

```
Открытый текст
   │
   ▼
┌─────────────────────┐
│  Plain Data Encoder  │  Сжатие текста в байты (лучший кодировщик выбирается автоматически)
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│       Pack          │  Добавление channel ID + timestamp + metainfo + HMAC
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│  Паддинг PKCS#7     │  Дополнение до границы 16 байт
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│    AES-256-GCM      │  Шифрование производным ключом
└──────────┬──────────┘
           ▼
┌─────────────────────┐
│ Encrypted Data      │  Кодирование шифротекста в Base64 / Кириллический Base64 /
│ Encoder             │  Русские слова
└──────────┬──────────┘
           ▼
      Текст SMS
```

## Вывод ключей

Единый **мастер-ключ** (128, 192 или 256 бит) разделяется между сторонами. Два подключа выводятся через HKDF (RFC 5869) с HMAC-SHA256:

```
HKDF-Extract:
  salt = "k-sms-hkdf-v2" (фиксированная, публичная)
  IKM  = master_key
  PRK  = HMAC-SHA256(salt, IKM)

HKDF-Expand (ключ шифрования):
  info   = "k-sms-v2-enc"
  length = 32 байта
  enc_key = HKDF-Expand(PRK, info, 32)

HKDF-Expand (ключ MAC):
  info   = "k-sms-v2-mac"
  length = 32 байта
  mac_key = HKDF-Expand(PRK, info, 32)
```

Salt и info — фиксированные константы протокола, не секреты. Они обеспечивают доменное разделение между ключами шифрования и MAC.

## Кодирование открытого текста

Перед шифрованием текст сообщения сжимается в массив байтов с помощью наиболее компактного доступного кодировщика. Кодировщик выбирается автоматически на основе содержимого:

| Режим | ID | Описание |
|---|---|---|
| SHORT_CP1251_PREFER_CYRILLIC | 0 | Компактная CP1251, оптимизирована для кириллицы |
| SHORT_CP1251_PREFER_LATIN | 1 | Компактная CP1251, оптимизирована для латиницы |
| CP1251 | 2 | Стандартная CP1251 |
| UTF_8 | 3 | Стандартный UTF-8 (запасной для любого текста) |
| ASCII | 4 | 7-битный ASCII |
| HUFFMAN_CYRILLIC | 5 | Код Хаффмана, оптимизирован для кириллицы |
| HUFFMAN_LATIN | 6 | Код Хаффмана, оптимизирован для латиницы |

ID кодировщика сохраняется в байте MetaInfo (см. ниже), чтобы получатель знал, как декодировать.

## Упаковка сообщения

Закодированные байты текста упаковываются в payload с метаданными:

```
┌──────────────┬─────────────────┬───────────────┬──────────┬──────────────┐
│ encoded_text │ channel_id (4Б) │ timestamp (4Б)│ meta (1Б)│ hmac (8Б)    │
│ (переменная) │ (опционально)   │               │          │              │
└──────────────┴─────────────────┴───────────────┴──────────┴──────────────┘
                                                            ▲
                                        HMAC покрывает ─────┘
                                  encoded_text + channel_id + timestamp
```

### Поля

**encoded_text** (переменная длина): Текст сообщения, сжатый кодировщиком.

**channel_id** (4 байта, опционально): Идентификатор канала диалога. Присутствует только если установлен флаг `isChannel` в MetaInfo. Little-endian uint32.

**timestamp** (4 байта): Unix timestamp в секундах, little-endian uint32. Используется для защиты от replay-атак.

**MetaInfo** (1 байт): Битово-упакованный байт метаданных:

```
Раскладка битов: [C][VVV][MMMM]

Биты 0-3 (MMMM): Режим кодировщика текста (0-6)
Биты 4-6 (VVV):   Версия протокола (сейчас 3)
Бит 7 (C):        Флаг наличия channel ID (0 = нет, 1 = да)
```

**HMAC** (8 байт): Усечённый HMAC-SHA256 по `encoded_text + channel_id + timestamp`, вычисленный с `mac_key`. Хранятся только первые 8 байт из полных 32 для экономии места в SMS.

### Паддинг PKCS#7

После упаковки payload дополняется до кратного 16 байт по PKCS#7:

- Длина паддинга = `16 - (длина_payload % 16)`
- Каждый байт паддинга равен длине паддинга (1-16)
- Всегда добавляется минимум 1 байт

Это скрывает точную длину сообщения от анализа трафика.

## Шифрование

Дополненный payload шифруется **AES-256-GCM**:

```
nonce = SecureRandom(12 байт)
ciphertext || tag = AES-256-GCM(enc_key, nonce, padded_payload)
```

Формат зашифрованных данных:

```
┌──────────────┬──────────────────────────┐
│ nonce (12Б)  │ ciphertext + GCM tag     │
│              │ (tag = 16 байт)          │
└──────────────┴──────────────────────────┘
```

Свойства:
- **Nonce**: 12 байт (96 бит), случайно генерируется через `SecureRandom` для каждого сообщения
- **GCM tag**: 128 бит (16 байт), обеспечивает аутентифицированное шифрование
- **Шифр**: `AES/GCM/NoPadding` (javax.crypto, аппаратно ускоренный на Android)

GCM обеспечивает одновременно конфиденциальность и целостность. Любое изменение шифротекста, nonce или тега приведёт к ошибке дешифрования.

## Кодирование шифротекста (стеганография)

Зашифрованный массив байтов кодируется в текстовую строку для передачи через SMS. Доступны три схемы:

### Base64 (Схема 0)

Стандартное Base64-кодирование. Компактно, но визуально очевидно как закодированные данные.

```
Вход:  [0xDE, 0xAD, ...]
Выход: "3q0t..."
```

### Кириллический Base64 (Схема 1)

Base64, где латинский алфавит заменён кириллическими символами:

```
Латиница:  ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/=
Кириллица: АБВГДЕЁЖЗИЙКЛМНОПРСТУФХЦЧШЩЫЬЭЮЯабвгдеёжзийклмнопрстуфхцчшщъыьэюя
```

Вывод выглядит как кириллический текст для случайного наблюдателя.

### Русские слова / Текст (Схема 2)

Кодирует зашифрованные байты как естественно выглядящий русский текст с помощью системы суб-кодировщиков:

- **Словарные суб-кодировщики**: Отображают значения байтов в русские слова из словарей
- **DateTime суб-кодировщик**: Кодирует значения как строки даты/времени
- **Суб-кодировщик пунктуации**: Кодирует значения как знаки препинания

Кодирование трактует зашифрованные данные как большое целое число и выполняет разложение в смешанной системе счисления по доступным суб-кодировщикам. Случайный байт переднего паддинга добавляется при необходимости, чтобы число корректно отображалось в доступное пространство слов.

Вывод выглядит как последовательность русских слов с естественной пунктуацией и пробелами. При декодировании удаляется до 256 байтов переднего паддинга для нахождения правильного выравнивания.

## Защита от replay-атак

Каждое сообщение содержит 4-байтовый Unix timestamp. При дешифрации timestamp валидируется:

- Сообщения старше **24 часов** отклоняются
- Сообщения из будущего более чем на **5 минут** отклоняются

Это предотвращает replay-атаки, когда перехваченный шифротекст отправляется повторно позже.

## Проверка HMAC

Усечённый HMAC проверяется с помощью **сравнения за константное время** (`MessageDigest.isEqual()`). Это предотвращает атаки по времени, которые могут утечь информацию об ожидаемом значении HMAC.

HMAC вычисляется по открытому payload (до шифрования), обеспечивая дополнительный слой аутентификации внутри шифротекста GCM. Хотя GCM уже обеспечивает аутентификацию, внутренний HMAC:

1. Аутентифицирует упакованную структуру (текст + метаданные) независимо
2. Позволяет раннее отклонение повреждённых данных после дешифрации, но до полного разбора

## Хранение ключей (Android)

### Ключи шифрования сообщений

Мастер-ключи для шифрования сообщений хранятся через **EncryptedSharedPreferences** (AndroidX Security):

- Мастер-ключ: AES-256-GCM (создаётся через Android Keystore)
- Шифрование ключей: AES-256-SIV
- Шифрование значений: AES-256-GCM

Ключи могут быть установлены глобально (для всех диалогов) или для каждого диалога отдельно (хранятся в зашифрованной базе Realm).

### Ключ базы данных Realm

База данных Realm зашифрована 512-битным ключом:

1. Генерируется случайный 64-байтовый ключ через `SecureRandom`
2. В **Android Keystore** (аппаратный TEE) создаётся ключ AES-256-GCM
3. Ключ Realm шифруется ключом из Keystore
4. Зашифрованный ключ Realm и IV сохраняются в SharedPreferences
5. При каждом запуске приложения ключ Realm расшифровывается ключом из Keystore

```
┌────────────┐     AES-256-GCM      ┌─────────────────────┐
│  Android   │ ───────────────────►  │  SharedPreferences   │
│  Keystore  │   encrypt(realm_key)  │  (зашифр. ключ + IV) │
│  (TEE)     │ ◄─────────────────── │                      │
│            │   decrypt(realm_key)  │                      │
└────────────┘                       └─────────────────────┘
```

## Обмен ключами

Ключи обмениваются **вне основного канала** между пользователями:

1. **QR-код**: Один пользователь генерирует QR-код с Base64-кодированным ключом; второй сканирует его
2. **Ручной ввод**: Пользователи копируют/вставляют Base64-кодированный ключ

Подлинность ключа можно проверить сравнением **SHA-256 отпечатков** (первые 16 байт, отображаемые в hex):

```
Байты ключа → SHA-256 → первые 16 байт → "A1 B2 C3 D4 E5 F6 ... "
```

Обе стороны должны сверить отпечаток через отдельный защищённый канал.

## Свойства безопасности

| Свойство | Механизм |
|---|---|
| Конфиденциальность | AES-256-GCM |
| Целостность | GCM authentication tag (128 бит) + HMAC-SHA256 (усечённый, 64 бит) |
| Защита от replay | 4-байтовый timestamp с 24-часовым окном |
| Разделение ключей | HKDF с различными info-строками |
| Хранение ключей | Android Keystore (TEE) + EncryptedSharedPreferences |
| Скрытие длины | Паддинг PKCS#7 до границы 16 байт |
| Стеганография | Base64 / Кириллический Base64 / Русские слова |
| Защита от timing-атак | Сравнение HMAC за константное время |
| Защита от повтора nonce | 12-байтовый случайный nonce через SecureRandom на каждое сообщение |

## Ограничения

- **Нет forward secrecy**: Компрометация мастер-ключа раскрывает все прошлые и будущие сообщения
- **Нет ротации ключей (ratcheting)**: Один мастер-ключ используется для всех сообщений в диалоге
- **Усечённый HMAC**: 64-битный HMAC обеспечивает стойкость к коллизиям 2^64 (достаточно для SMS, но ниже типичного порога в 128 бит)
- **Точность timestamp**: Разрешение 1 секунда, 32-битный Unix timestamp (переполнение в 2106 году)
- **Ограничения размера SMS**: Стеганографическое кодирование увеличивает размер сообщения; длинные сообщения могут разбиваться на несколько SMS-сегментов оператором
- **Нет отрицаемости**: Обе стороны используют один симметричный ключ и могут доказать, что другая сторона отправила сообщение
